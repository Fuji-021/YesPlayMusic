# PodPlayer 多实例数据隔离规范（事故根治）

> 来源：`调查报告-IndexedDB无法打开.md` 的根治方案。本次 `backing store 打不开` 的根因是**多个同身份程序共用同一个 IndexedDB 库、抢 LevelDB 独占锁**。本规范从根上消除它：让每个实例拥有**独立身份 → 独立 userData → 独立数据库**，三者永不共库、可同时运行、互不污染。
> 一句话原理：**`app.setName(X)` 决定 `userData = %APPDATA%\X\` 决定 IndexedDB 物理位置 `…\X\IndexedDB\`。只要 name 唯一，库就物理隔离。**

## 一、三个实例的定义（彻底隔离）

| 实例 | 用途 | app name（决定库位置） | userData 目录 | 窗口标题/安装名 |
|---|---|---|---|---|
| **测试床** | 跑种子/测试数据、试破坏性操作、回归验证 | `PodPlayerSandbox` | `%APPDATA%\PodPlayerSandbox\` | `PodPlayer 测试床` |
| **开发版** | 主 session 日常开发（dev-serve / 开发构建） | `PodPlayerDev` | `%APPDATA%\PodPlayerDev\` | `PodPlayer Dev` |
| **正式版** | 你日常真正使用（安装包） | `PodPlayer` | `%APPDATA%\PodPlayer\` | `PodPlayer` |

→ 三个 name 互不相同 → 三个 `…\IndexedDB\` 互不相同 → **三者同时开也各读各库、永不抢同一 LOCK**，根治 backing store 锁争用；且在测试床里清库/试错，**绝不会动到开发版或正式版的数据**。

> 你的诉求"dev 改名叫测试床"= 测试床实例的 name/标题/安装名统一带"测试床/Sandbox"，肉眼一眼可辨，避免在错的实例上误操作（这次事故的人为诱因之一）。

## 二、端口分配（每实例一套，避免端口漂移/冲突）

端口冲突会导致 dev server 换端口 → 换 origin → 连到另一个库（数据"看似丢失"）。故每实例固定独立三件套：

| 实例 | vue dev 端口 | 网易云 API 端口 | 内部 express 端口 |
|---|---|---|---|
| 测试床 | 20202 | 10756 | 27234 |
| 开发版 | 20201 | 10755 | 27233 |
| 正式版 | （打包无 dev server） | 10754 | 27232 |

## 三、运行规则（五条铁律）

1. **物理隔离为本**：靠"独立 name → 独立 userData → 独立库"实现隔离，**不靠纪律靠机制**。这是根本保证。
2. **独立端口**：每实例固定自己的三件套端口，杜绝端口被占→漂移→换 origin→连错库。
3. **单实例锁**：同一实例只允许开一个（`requestSingleInstanceLock`，第二个直接退出并前置已有窗口）；**不同实例可并存**（因库已隔离，无害）。
4. **启动脚本只清"本实例"**：按窗口标题/profile 精确清理本实例残留进程，**不得 `taskkill electron.exe` 一锅端**（会误杀别的实例）。
5. **可视化区分**：窗口标题、托盘提示都带实例名（"PodPlayer 测试床"），并可给测试床/开发版换个角标或主题色，避免在错的实例上做操作。

## 四、落地方式（交主 session；与修复 background.js 截断一并做）

用一个环境变量 `PODPLAYER_PROFILE` 驱动一套代码跑三实例（最干净）：

```js
// background.js（修复截断时加入；放在 setName 之前）
const path = require('path');
const PROFILE = process.env.PODPLAYER_PROFILE || 'prod';
const PROFILES = {
  prod:    { name: 'PodPlayer',        title: 'PodPlayer',        neapi: 10754, express: 27232 },
  dev:     { name: 'PodPlayerDev',     title: 'PodPlayer Dev',    neapi: 10755, express: 27233 },
  sandbox: { name: 'PodPlayerSandbox', title: 'PodPlayer 测试床', neapi: 10756, express: 27234 },
};
const P = PROFILES[PROFILE] || PROFILES.prod;
app.setName(P.name);
app.setPath('userData', path.join(app.getPath('appData'), P.name));
// 网易云 API / express 监听用 P.neapi / P.express；窗口 title 用 P.title
```

配套：
- **vue serve 端口**：每实例一份 `.env`（`.env.sandbox` / `.env.development`）设 `VUE_APP_*` 与 devServer 端口（20202 / 20201），或在启动脚本里传 `--port`。渲染端连后端用对应 `PODPLAYER_PROFILE` 的端口。
- **启动脚本**（各一个，互不干扰）：
  - `start-sandbox.bat`：`set PODPLAYER_PROFILE=sandbox` + vue serve `20202` + electron；清理只针对测试床窗口/端口。
  - `start-dev.bat`：`set PODPLAYER_PROFILE=dev` + `20201`；**进程清理改为按本实例**（勿再 `taskkill electron.exe` 全杀）。
- **打包**：正式版 `productName: PodPlayer`、`PODPLAYER_PROFILE=prod`；测试床如需出独立安装包则 `productName: PodPlayer 测试床` + `appId` 独立 + `PODPLAYER_PROFILE=sandbox`。

## 五、验收（达成即根治）

1. **三实例可同时启动**，各自正常打开、互不报 `backing store` 错。
2. 文件验证：`%APPDATA%` 下出现三个独立目录 `PodPlayer\`、`PodPlayerDev\`、`PodPlayerSandbox\`，各自有独立的 `IndexedDB\…leveldb`。
3. 在**测试床**里删库 / 清数据 / 试破坏性操作 → 开发版、正式版数据**完全不受影响**。
4. 关闭其中任一实例（含异常残留进程）→ 不影响其余实例打开。
5. 端口固定：每实例始终在自己的端口，不再因占用而漂移。

## 六、迁移注意

- 改 name 后，旧 `%APPDATA%\YesPlayMusicPodcast\` 里的数据**不会自动搬到**新目录（库按 name 隔离）。若旧库还有可救数据（先按调查报告 §6.2 确认非损坏），可手动把旧 `…\YesPlayMusicPodcast\IndexedDB\` 整目录拷到新实例对应目录后再重命名匹配；否则按调查报告 §7 走 OPML 重灌。
- 本规范落地前提是 `background.js` 截断损坏已修复（见调查报告 §8 第 1 条）。
