# 可探索资源池方案 —— 丰富首页内容 / 撑起搜索与推荐

> 2026-06-11 ｜ 性质：功能规划（非 bug）｜ 结论均注明"读码确认 / 联网核实 / 实测验证 / 待实测"四种证据等级
> 诉求：当前获取资源途径有限（首页只能依赖 iTunes 体系，部分失效）；评估小宇宙等权威资源池的可行性，目标同时喂饱**首页内容、搜索、为你推荐**三个功能。

## 一、现状与真实痛点（读码确认）

当前链路（`electron/podcastDiscover.js` + `utils/podcast/discover.js`）：

| 环节 | 现状 | 痛点 |
|---|---|---|
| 榜单 | xyzrank 双榜（`/api/podcasts` + `/api/new-podcasts`，需带 Referer 反反爬） | 池子单点：≤250 条 → B67-BUG-3 推荐池枯竭（"为你推荐只剩 3 个"的根子） |
| feedUrl 解析 | **仅** Apple `itunes.apple.com/lookup` | 解析单点：xyzrank 条目无 Apple 链接即死（《她山石》/B-70 红点） |
| 搜索 | **仅** iTunes Search | 搜索单点 |

三个痛点本质相同：**每个环节都是单点**。方案目标 = 每个环节至少两条腿。

## 二、小宇宙：结论是不做主源（联网核实，2026-06）

1. **无官方开放 API**。社区只有逆向方案（[ultrazg/xyz](https://github.com/ultrazg/xyz)、xiaoyuzhoufm-mcp、xyz-dl 等），全部需**手机号+验证码登录拿 token**（refresh_token + device_id），接口随 App 版本漂移、违反 ToS、有封号风险。
2. **架构不兼容（更致命）**：本播放器是 RSS 架构，一切源最终要落到 feedUrl；小宇宙自建托管、**不对外输出 RSS**——爬到它的榜单后，每个节目仍要回头"按名字找 RSS"，等于绕路。
3. **已经间接拥有它**：xyzrank 本身就是聚合中文播客全平台（含小宇宙）热度的榜单，保留 xyzrank 即保留小宇宙的热度信号。
4. 可选兜底（灰色、按需）：[RSSHub](https://github.com/DIYgod/RSSHub) 有小宇宙路由可把节目页转 RSS（建议自建实例），仅作"全网找不到 RSS"节目的最后通道，稳定性一般，不进主链路。

## 三、推荐接入的两条权威开放源

### A. Apple 官方榜单 JSON（零鉴权、官方稳定 —— ✅ 已实测验证，2026-06-11）

**实测**：`https://rss.marketingtools.apple.com/api/v2/cn/podcasts/top/{N}/podcasts.json` 直接返回**中国区中文播客热门榜**——样本前 10：岩中花述 / 自我进化论 / 声动早咖啡 / 凹凸电波 / 搞钱女孩 / 谭立人 / 天真不天真 / 知行小酒馆 / 霓达播客 / 肥话连篇，全部是中文头部节目，且每条带 **Apple id**（与现有 lookup 解析无缝衔接）、**中文类目体系**（艺术/商务/喜剧/社会与文化…genreId）和封面 URL。
⚠️ 同时实测：**旧版端点 `itunes.apple.com/cn/rss/toppodcasts/.../json` 已返回空**，不要用旧端点（这也回应了"Apple 开放目录在收紧"的背景，见 [HN 讨论](https://news.ycombinator.com/item?id=41310647)）。
生成器/参数参考：[rss.marketingtools.apple.com](https://rss.marketingtools.apple.com/)。

**回答"有没有中文节目"**：有，且就是中文圈头部内容（上面 10 个名字即证据），并自带中文分类目，首页可按类目铺多板块（科技榜/社会文化榜/商业榜…）。

### B. Podcast Index（免费 key，根治"解析单点"与"池子枯竭"）

[podcastindex.org](https://api.podcastindex.org/)：开放播客索引（400 万+ feed），**免费注册即得 API key**（[文档](https://podcastindex-org.github.io/docs-api/)），数据库甚至可整库下载。能力与痛点一一对应：

| 端点 | 喂给哪个功能 |
|---|---|
| `/search/byterm`（**直接返回 feedUrl**） | 搜索第二源（与 iTunes 合并去重）；**feedUrl 解析链第二级**——无 Apple 源时按名字搜，《她山石》类失败大减、B-70 红点自愈 |
| `/podcasts/trending?lang=zh` | 首页新板块 + 为你推荐池从 ≤250 扩到数千（B67-BUG-3 根治） |
| `/categories` + 按类目 trending | 推荐算法"搜索词类目加权"获得真实类目体系 |
| feed 状态字段（dead / lastHttpStatus） | 失效源**事前预警**（红点从"事后记忆"升级） |

**回答"有没有中文节目"（如实标注）**：机制上，凡有公开 RSS 的中文播客（= 能上 Apple 的那批，含你订阅库里的全部节目）都在其收录范围，且 trending 支持 `lang=zh`；但**中文覆盖率与中文搜索质量我没有第一手实测**（API 要 key，无 key 的两条路径已试均不可达）。建议先做 30 分钟 spike：注册 key → 拿 xyzrank top100 节目名跑 `/search/byterm` → 统计命中率。命中率 ≥80% 则按本方案权重接入；明显偏低则降级为"仅解析链第二级 + 英文/泛资源板块"。

### C/D. 其他

- **xyzrank 保留**：中文热度信号最准（聚合小宇宙等平台），继续当中文热度特色源。
- **商业 API**（Listen Notes / Podchaser）：搜索与推荐质量最高但收费、免费额度有限，现阶段不必，记录备查。

## 四、落地架构（与现有代码衔接）

1. **Source Adapter 模式**：`src/utils/podcast/sources/{xyzrank,appleCharts,podcastIndex}.js`，统一归一化输出 `{ name, author, coverUrl, feedUrl?, appleId?, hot, source }`；网络请求仍走主进程 ipc（PodcastIndex 的 key/secret 放主进程，沿用 podcastDiscover.js 的 axios+缓存套路）。
2. **feedUrl 解析链**（discover.js `resolveFeed` 改造，对 B-70 直接增益）：`自带 feedUrl → Apple lookup → PodcastIndex byterm 按名匹配 → (可选)RSSHub 小宇宙路由 → 标红点`，每级命中结果缓存进 Dexie。
3. **去重合并**：多源按 feedUrl（首选）/规整化节目名（兜底）去重，扩展现有三栏去重链（B-64 #12 usedNames 机制）。
4. **限频与缓存**：榜单沿用 `podcast:fetchHot` 缓存策略（force 才刷）；Apple 榜/PodcastIndex trending 每日 1 次足够；搜索按词缓存 10 分钟。
5. **实施顺序**：
   ① Apple 榜单 adapter（~40 行，已实测可用，零风险先吃收益）
   ② PodcastIndex spike 实测中文命中率（30 分钟，定权重）
   ③ PodcastIndex adapter + 解析链第二级（~120 行，核心收益）
   ④ 搜索双源合并（~40 行）
   ⑤ 推荐池接 trending/Apple 类目榜（复用 buildForYou，池大后 B67-BUG-3 自然消解）
   ⑥ RSSHub 小宇宙兜底（可选）

> 定位总结：xyzrank 管"中文热度"，Apple CN 榜管"官方稳定 + 类目多样"，PodcastIndex 管"广度 + 解析 + 搜索"。三源各司其职，三个单点全部消除。

## 五、接入性实测（2026-06-12，只读 spike，未改代码）

> 目的：在不接入的前提下验证"能不能接"。全部用 curl / Invoke-WebRequest 直打端点，模拟主进程 axios 行为。

| 测试项 | 结果 | 证据 |
|---|---|---|
| Apple 新榜 `top/10/podcasts.json` | ✅ 200 | 中文头部内容（岩中花述等），每条带 `id`/`name`/`artistName`/`artworkUrl100`/`genres`(中文类目)/`url` |
| Apple 新榜容量上限 | **100** | top/100=200 返回 100 条；top/200 → 500 错误。推荐池扩容要靠"多类目榜并集"或 PodcastIndex，单榜最多 100 |
| **全链路**：榜单 id → lookup → RSS | ✅ 全通 | id=1582119137 → `itunes.apple.com/lookup` → `https://feed.xyzfm.space/hwen8wf69c6g` → 抓 RSS 200（482KB，82 item 全带 enclosure）。**现有 resolveFeed + parseRss 零改造可消化** |
| 旧版端点 `itunes.apple.com/cn/rss/toppodcasts` | ⚠️ 今日实测 200 有数据（17KB，含岩中花述） | **修正第三节"已返回空"的说法：是间歇性失效，不是死了**。仍用新端点为主，旧端点可记为零成本兜底 |
| PodcastIndex 无 key 探测 | 符合预期 | 裸 UA → 403（要求自定义 UA）；带 UA → 401（缺 Authorization）。API 活着，鉴权方案与文档一致（sha1(key+secret+unixtime)，Node 内置 crypto，**零新依赖**）。中文命中率必须注册 key 后才能测 |
| xyzrank 双榜 | ✅ 200 | hot 52KB / new 44KB，现有源今日健康 |
| iTunes Search 中文词 | ✅ 200 | "知行小酒馆"返回 feedUrl，现有搜索源今日健康 |

**意外发现**：岩中花述的 feedUrl 域名是 `feed.xyzfm.space`——即**小宇宙托管的节目其实对外输出 RSS**（用于分发到 Apple）。第二节"小宇宙不输出 RSS"需精确化为：平台独占/会员内容不输出，但其托管的公开节目通过 Apple 目录可正常拿到 RSS。这进一步支持"不逆向小宇宙"的结论：公开内容已经间接可达。

**安全评估**：Apple 端点零鉴权官方域名，无风险；PodcastIndex key/secret 放主进程本地配置即可，唯一注意点是**不要把 key 提交进 git**（建议本地 json/.env + gitignore）；全程零新 npm 依赖，不碰 yarn.lock；RSS 内容沿用现有 sanitizeHtml 链路。结论：①②③④⑤ 步均可低风险落地。

**下一步唯一阻塞项**：注册 PodcastIndex 免费 key（api.podcastindex.org，邮箱即可）→ 跑第三节 B 的 30 分钟中文命中率 spike 定权重。
