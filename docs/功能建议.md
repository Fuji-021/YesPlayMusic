## 建议 ① NAS 配置中心（设置页）—— 来源：R10 审查发现 P2 未落地 + 2026-06-13 用户诉求

### 背景与诉求

R10 审查坐实 NAS 音源已真接入并工作，但 `settings.vue` **无任何 NAS 配置 UI**（命中=0）；当前配置靠手写 `nas-config.json`。用户诉求（原话提炼，权威）：

1. **NAS 接口可配置**——换了 NAS / 多个 NAS 之间能重新配置、切换；
2. **简单好操作**——不要一步步手填那些难记的参数（尤其库的 UUID）；
3. **连接历史 + 一键连接**——连接过的存成历史，点一下就重连，不必每次重输。

> 本质：把"单份静态配置"升级成"**多档连接 + 历史 + 一键切换**的配置中心"。

### 数据模型（`nas-config.json`，electron-store，**token 仍只在主进程、不入 git**）

```jsonc
{
  "enabled": false,            // 总开关：是否启用 NAS 就近音源（默认关，双保险沿用）
  "activeProfileId": "",       // 当前激活的连接档 id
  "profiles": [
    {
      "id": "uuid",
      "name": "家里飞牛",       // 备注名，默认用 host:port
      "baseUrl": "http://192.168.2.108:13378/audiobookshelf",
      "token": "***",          // 仅主进程读写；渲染端只拿到 hasToken
      "libraryId": "c4379a59-…",
      "libraryName": "podcast", // 显示用，发现库时一并存
      "lastConnectedAt": 0,
      "createdAt": 0
    }
  ]
}
```

向后兼容：启动时若检测到旧的扁平配置（`baseUrl/token/libraryId` 在顶层），自动迁移成 `profiles[0]` 并设为 active。

### 简单好操作的三个关键设计

1. **自动发现库，免填 UUID（最重要的体验点）**：用户**只填地址 + token**，点「测试并发现库」→ 主进程拿 baseUrl+token 调 `GET /api/libraries` → 过滤出 podcast 类型库 → 前端**下拉选**。那串 `c4379a59-72a4-…` 的库 id 由此自动取得，用户从不手敲。（这是当前手写配置里最痛的一步。）
2. **一键连接历史**：历史档列表每行一个「连接」按钮，点击 = 设为 active + 重探 + `initNas()`，秒切，不重输。换 NAS、出差回家切不同实例都顺手。
3. **存前先测**：保存档前必须「测试连接」通过（`/ping` + 库可列），状态点实时绿/红反馈，避免存进死配置。

### 设置页 UI（`settings.vue` 新增「NAS 就近音源」区块，沿用项目设置页风格）

```
┌ NAS 就近音源 ───────────────────────────────┐
│ [开关] 启用 NAS 就近音源                       │  ← 总开关，关=②级解析链不激活（=现状）
│                                               │
│ 当前连接： ● 家里飞牛   192.168.2.108:13378    │  ← ● 绿/红状态点(复用 nasStatus)
│           库：podcast        [测试连接]        │
│                                               │
│ 连接历史                                       │
│  ○ 家里飞牛   …108:13378   3 分钟前  [连接][✎][🗑]│  ← 一键连接 / 编辑 / 删除
│  ○ 公司NAS    …50:13378    上周      [连接][✎][🗑]│
│                                               │
│ [ + 添加连接 ]                                 │
└───────────────────────────────────────────────┘

「添加/编辑连接」弹层（原位小弹窗，沿用项目弹窗风格，非全屏）：
  名称(可选)  [____________]
  地址        [http://192.168.2.108:13378/audiobookshelf]
  Token       [____________]  （提示：ABS→设置→用户→API Token 复制）
  [ 测试并发现库 ] → 成功后出现：  库 [下拉：podcast ▾]
  [ 保存 ]
```

### 主进程 IPC 增补（`nasBridge.js`）

| IPC | 入参 | 作用 |
|---|---|---|
| `nas:listLibraries` | `{baseUrl, token}` | 用**临时**凭据调 `/api/libraries`，返回 `[{id,name,mediaType}]` 过滤 podcast → 供"发现库"下拉（测试用，未必已保存） |
| `nas:listProfiles` | — | 返回 profiles（**去 token**，只回 hasToken/name/baseUrl/libraryName/lastConnectedAt） |
| `nas:saveProfile` | `{profile}` | 新增/更新一档（含 token，主进程存） |
| `nas:deleteProfile` | `{id}` | 删除一档 |
| `nas:activateProfile` | `{id}` | 设为 active + 清缓存 + 更新 lastConnectedAt；渲染端随后 `initNas()` 重探 |

现有 `nas:getStatus/setConfig/probe/resolve/...` 改为**读 activeProfile**（其余逻辑不变）。`getCfg()` 内部从 `profiles.find(activeProfileId)` 取，对上层透明。

### 实施量与风险

- 渲染端：settings.vue 一个区块 + 添加/编辑弹层（~150 行）；复用现有 `setNasConfig/testNasConnection/nasStatus`。
- 主进程：`nasBridge.js` 加 5 个 IPC + `getCfg` 改 active 取档（~80 行）；新增 `nas:listLibraries` 是唯一新网络调用（`/api/libraries`，已在 R10/P0 实测可用）。
- 风险低：不碰解析链①②③与播放；总开关关=现状；token 不出主进程口径不变。**属方案原 P2 的落地 + 用户体验增强，可独立成一个 commit。**

## 建议 ② NAS 断联补强（播放中掉线无缝续播）—— 来源：R10 抓到的 B-NAS-2 缺口

### 现状缺口（读码 R10）

`Player.js:513` 只监听 `loaderror`。NAS 播到一半若抛 `loaderror`，会走既有"重建+seek"路径、经三级链自动落 CDN（OK）。**但"播放中数据流悄悄停住、不抛 error"的静默 stall 不会被发现 → 卡住不动。** 方案 P3 原本要求 loaderror + playerror 双覆盖，当前只做了前者。

### 真实案例参考（联网核实 2026-06-13）

- **HTML5 媒体标准信号**：`stalled`（媒体数据获取意外停止）、`waiting`（因缺数据暂停等待）、`error`——这是浏览器检测"数据没来"的标准事件，所有流媒体播放器都建在其上。
- **HLS.js 的成熟范式**（[error recovery](https://github.com/video-dev/hls.js/issues/813) / [bufferStalled](https://github.com/video-dev/hls.js/issues/2893)）：`bufferStalledError` 检测卡顿 → 先 `bufferNudgeOnStall` 轻推自愈 → 仍不行 `recoverMediaError()`（**经验：恢复前加一小段 `setTimeout` 延迟**，否则抖动）→ 致命错误才切源。**先宽限自愈、再 failover** 是关键心法，避免对瞬时缓冲抖动过度反应。
- **howler 的局限（context7 查证）**：howler 在 `html5:true` 流式下**不透传** `stalled/waiting`，只有 `loaderror/playerror`。要拿这些标准信号需取底层 `<audio>` 节点：`Howl._sounds[0]._node`（howler 未公开但稳定的内部结构，社区常用——依赖项，需标注）。

### 方案（仅对 NAS 源激活；CDN/本地源路径一字不动）

**两层检测 + 宽限 failover**：

1. **事件层**：播 NAS 源时，取 `this._howler._sounds[0]._node` 挂原生 `stalled`/`waiting`/`error` 监听（标准信号、即时）。
2. **看门狗层（兜底，不依赖 howler 内部）**：复用现有每秒 tick——`playing` 状态下 `seek()` 返回值连续 ~8s 不前进 → 判定静默 stall。即使事件不触发也能抓住。
3. **宽限自愈**（仿 HLS.js）：检测到 stall 后**先等 2–3s** 给缓冲自愈机会（NAS 短暂抖动会自己恢复，别误切）；仍不动才 failover。
4. **failover 续播**：记录当前 `progress` → **强制走 ③CDN 重建 Howl**（给 `_getAudioSource` 加 `forceCdn` 参数跳过②，否则会重新解析 NAS 又卡）→ `once('load')` 后 `seek(progress)` 续播 → 同时置 `nasAlive=false`（本集及随后切集先走 CDN，由 5min 心跳探活恢复）。
5. **防抖限次**：每集最多 failover 1–2 次，避免 NAS↔CDN 抖动循环；与既有 `loaderror` 路径合并（同一个"重建+seek"函数，避免两套）。

### 验收

- Dev 真机：播一个 NAS 源单集 → 中途**停掉 ABS 容器** → 预期 2–3s 宽限后自动落 CDN 续播、进度误差 <3s、toast/状态点反映音源切换；不再卡死。
- 对照：CDN/本地源播放路径行为与修前完全一致（补强仅对 NAS 源激活）。

## 建议 ③ NAS 来源呼吸灯：齐步 → 萤火虫式柔和呼吸（错相位 + 微变周期 + 柔和曲线）—— 来源：2026-06-13 用户讨论 + 联网核实成熟设计

### 现状（读码确认）

NAS 来源标识是**纯 `opacity` 呼吸**（合成器动画、不触发重排重绘，最省的一类），三处共用一套 `@keyframes nas-breathe`（`0%/100% opacity:1 → 50% opacity:0.45`，`2.6s ease-in-out infinite`、**无 `animation-delay`**）：

| 位置 | 元素 | 同屏数量 |
|---|---|---|
| 单集列表点 | `podcastDetail.vue` `.nas-dot`（:106；样式 :1227/keyframes :1241） | **多**（每 NAS 单集一个，可能上百） |
| 订阅卡片点 | `podcastLibrary.vue` `.nas-dot`（:171；样式 :1451） | **多**（每 NAS 节目一个，几十） |
| 导航栏状态点 | `Navbar.vue` `.nas-status`（:35） | 1（单点） |

问题：① 无 delay → 齐步同呼吸（机械、像同步报警）；② 2.6s **偏快**（接近"心跳"而非"呼吸"）；③ 对称 ease-in-out → 不够"呼吸"的不对称柔和感。

### 成熟设计参考（联网核实 2026-06-13）

- **苹果睡眠呼吸灯（标杆）**：周期 ~**4–7s**（贴合人静息呼吸频率）；曲线是**两段不同高斯拼接**（非线性 + 亮起/熄灭**不对称**，因人眼对亮度变化非线性响应，高斯比 sine 自然）；**峰值只到 ~25% duty**（多数时间偏暗、不刺眼）。"舒服"的来源 = 慢 + 柔 + 不对称 + 不太亮。
- **萤火虫/柔和辉光（前端共识）**：opacity/scale 缓动 + **每个个体 `animation-delay` 与 `animation-duration` 都略不同** → 群体此起彼伏、缓慢飘移；**纯 CSS、无 JS**；性能优化要点 = 只动 opacity/transform（合成器），别逐帧动 box-shadow 的 blur。

### 推荐：三个杠杆，全部"纯 CSS + id 派生的确定值"（零 JS 运行时、零 bug）

回答"固定速度合适吗"：**单一固定速度欠一点萤火虫味**。但不需要真随机（那才有性能/闪烁 bug）——用"**每点固定、彼此不同**"的确定性多样即可：

1. **错相位**：负 `animation-delay`（id 派生）→ 群体不同步。
2. **微变周期**：`animation-duration` 取 **3–5s 区间的几个非谐波档**（id 派生）→ 个体节奏不同、缓慢飘移，这正是"固定速度不够"的补法。
3. **调慢 + 柔和不对称曲线**：基础周期 2.6s → **~3.6s**；曲线用 ease-out（简单）或分段 cubic-bezier 模拟高斯不对称（进阶）；opacity 谷别太低、峰别长留（偏暗时间多一点，仿 25% duty）。

导航栏 `.nas-status` 单点**不改**（就一个，人眼无从比对同步）。

### 精确改法（一个方法返回两个 id 派生值；keyframes 调柔；周期调慢）

```js
// methods（podcastDetail.vue 与 podcastLibrary.vue 各一份）
// 由稳定 id 派生"周期 + 负相位"两个确定值：同一点每次渲染恒定，彼此却不同 → 萤火虫式不齐
nasGlow(id) {
  let h = 0; const s = String(id || '');
  for (let i = 0; i < s.length; i++) h = (h * 31 + s.charCodeAt(i)) >>> 0;
  const dur = [3.2, 3.8, 4.4, 5.0][h % 4];                 // 4 档非谐波周期，缓慢飘移、极少全体重合
  const delay = -(((h >> 3) % Math.round(dur * 100)) / 100); // 负相位，铺满各自周期；用 h>>3 与周期解耦
  return { animationDuration: dur + 's', animationDelay: delay.toFixed(2) + 's' };
}
```

```html
<!-- podcastDetail.vue :106 / podcastLibrary.vue :171 -->
<span class="nas-dot" :style="nasGlow(ep.id)" …>   <!-- 订阅页用 nasGlow(p.id) -->
```

```scss
// keyframes 调柔 + 轻微不对称（升缓、峰短、降缓且占比更长，仿高斯/25% duty）；谷抬到 .5 更柔
@keyframes nas-breathe {
  0%   { opacity: .5; animation-timing-function: cubic-bezier(.45,0,.55,1); } // 升：缓
  40%  { opacity: 1; animation-timing-function: cubic-bezier(.45,0,.55,1); }  // 峰：偏早、短暂
  100% { opacity: .5; }                                                        // 降：缓、占 60%
}
// .nas-dot 基础周期调到 3.6s（会被 nasGlow 的 animationDuration 按点覆盖）：
// animation: nas-breathe 3.6s ease-in-out infinite;
```

### 避坑清单（满足"不加新 bug"，逐条对应真实陷阱）

1. **必须 id 派生（确定性），绝不用 `Math.random()`**：随机值会在列表重渲（排序/过滤/后台水合）时重新摇号 → 相位/周期突跳 → 闪烁 = 新 bug。
2. **由 id 派生而非渲染位置 index**：两处 `v-for` 都没取 index、`:key` 用稳定 id；id 派生则**排序也不移相/不变周期**。
3. **负 delay，别用正 delay**：负值让动画起步即在周期中段（错落立现）；正值=齐步等一会再一起开始。
4. **周期变化要"少档 + 非谐波 + 固定"**（修正本建议早期"别动 duration"的说法——那是针对"整齐"目标）：用 3.2/3.8/4.4/5.0 这类**比例不简单**的几档，缓慢飘移、几乎不会全体重合；**切忌连续随机 duration**（无规律 + 重渲重摇 = 抖动）。萤火虫要的正是"同区间、各不同、慢飘移"。
5. ⚠️ **兼容性：别用 `linear()` 缓动函数**。项目是 **Electron 13 = Chromium 91**，而 CSS `linear()`（用来精确拟合高斯曲线）是 **Chrome 113+** 才有 → 在 91 上**静默失效**回退成 linear，反而更生硬。用 `cubic-bezier()` / 分段 `animation-timing-function`（古老普遍支持）即可。
6. v-for 已按 id `:key`，DOM 不复用错位，无需额外处理。

### 性能 / bug 结论

零新增成本：仍是 `opacity` 合成器动画，`nasGlow` 只在渲染时按可见点算一次哈希（极廉价、无每帧 JS）；同屏多点开销与齐步版**完全相同**（变的只是各自相位/周期，动画层数不变）。bug 风险≈0（纯 CSS + 确定性派生，避开第 1、5 条即可）。可选加分项：`@media (prefers-reduced-motion: reduce)` 降为静态/慢速，照顾无障碍。

> 落地分两档：**必做**（收益最大、最简）= 调慢到 ~3.6s + `nasGlow` 错相位与微变周期；**选做**（锦上添花）= 不对称高斯曲线 + reduced-motion。纯渲染端、独立小改，可单独一个 commit。

## 附：监测职责分层（回答"要不要在 NAS 再加监测容器"——2026-06-13 用户问）

**不需要再加 NAS 监测容器，会冗余。** 两类"健康"在不同层、各归其位：

| 层 | 谁来管 | 管什么 | 跑在哪 |
|---|---|---|---|
| NAS 侧 ABS 运维 | 现有 `abs-watchdog` 容器 | 下载卡死自动重启+重新入队、存储超 500G 驱逐 | **NAS 上**（需 docker.sock 重启 ABS） |
| 客户端播放可达性 | 建议②（断联补强） | 听到一半 NAS 掉线/卡住 → 落 CDN 续播 | **PC 上的 app 里** |

关键：断联发生时 **NAS 本身已不可达**，NAS 上的任何容器都救不了"此刻这台 PC 的播放续播"——续播只能由客户端就地完成。所以断联补强是纯 app 层的事，与 NAS 容器无关；现有 watchdog 已覆盖 NAS 侧该管的全部，无需第二个监测容器。
