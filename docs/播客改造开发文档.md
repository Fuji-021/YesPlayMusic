## 1. 项目背景与目标

- **来源**：fork 自开源项目 [qier222/YesPlayMusic](https://github.com/qier222/YesPlayMusic)（MIT 协议），本地路径 `D:\MyYesPlayerMusic\YesPlayMusic`。
- **目标**：改造成**个人专用**桌面播客客户端，定位类似"桌面版小宇宙 / Apple Podcasts"。仅个人使用，高度个性化。
- **核心诉求**：导入订阅源（OPML 文件 / 单档 RSS URL）→ 在线或离线收听 → 保存每集播放进度。
- **复用**：整体布局、底部播放条、howler 播放、封面/列表组件、沉浸式展开页、Dexie 本地存储、路由前进后退、原有美术风格。
- **本阶段原则**：功能优先、能用为先；UI/美术打磨留到后续阶段，不引入新设计资源；工具链优先沿用项目原本要求的版本，老工具实在不行才换新。

### 需求三清单
- **保留复用**：窗口布局/缩放（Electron 自带）、顶栏 `< >` 前进后退、底部播放条按钮布局、播放控制逻辑、封面/列表组件、"点播放条展开沉浸页"机制、Dexie 数据库。
- **屏蔽（藏 UI，留源码）**：网易云登录、发现页、首页全部推荐板块、顶栏搜索、MV；歌词页保留入口（后续改造为"节目文稿"）。
- **新建**：订阅模块（RSS/OPML 抓取解析）、节目/单集数据模型、播放引擎改造（播已知音频直链）、按集保存进度、（后续）离线下载。

## 2. 技术架构速查

| 名词 | 作用 | 项目中的角色 |
|---|---|---|
| Vue 2 | 界面框架 | 全部界面（`.vue`） |
| Electron | 网页打包成桌面 App | 桌面壳、窗口管理 |
| howler.js | 音频播放引擎 | 真正出声的部件，播客复用 |
| Vue Router | 页面路由 | 导航、`< >` 前进后退 |
| Vuex | 全局状态 | 运行时状态，存 localStorage |
| Dexie/IndexedDB | 本地数据库 | 离线缓存，播客复用存订阅/节目 |

**关键文件（`src/` 下）**
- `utils/Player.js` ★★★ 播放引擎（class）。以网易云数字 id 为键：`_replaceCurrentTrack(id)`→`getTrackDetail(id)`取元数据→`_getAudioSource(track)`【缓存→`getMP3(id)`网易云API→UnblockMusic】→howler 播放。**改造关键**：让它能播 RSS 提供的已知音频直链，绕过 id 查询。
- `components/Player.vue` ★ 底部播放条 UI。点击空白→`toggleLyrics()`展开全屏 `views/lyrics.vue`（`store.showLyrics` 控制）。
- `views/home.vue` 首页（要做减法）、`views/explore.vue` 发现（屏蔽）、`views/library.vue` 音乐库（改造成播客库）、`views/lyrics.vue` 全屏沉浸页（复用为播客播放页）。
- `components/Navbar.vue` 顶栏（前进后退/导航/搜索/头像）。`go()` 用 `$router.go(±1)`，无网易云依赖，可原样复用。
- `router/index.js` 路由表 + 登录守卫（`requireLogin`）。
- `store/` Vuex 状态。`utils/db.js` Dexie 读写。`utils/request.js`+`api/*` 网易云请求（播客基本不用）。

**播放引擎适配（utils/Player.js 关键改动）**
原引擎工作流：`_replaceCurrentTrack(id) → getTrackDetail(id) → _getAudioSource(track) → _playAudioSource(url)`。其中 `_getAudioSource` 走 [缓存 → 网易云 getMP3(id) → UnblockMusic] 三级取源。播客没有 id、也没有网易云接口，但**只要 `_playAudioSource(url)` 能拿到音频地址，整条管子是通的**。所以做了三处微改：
1. `_getAudioSource(track)` 开头：`if (track.podcastAudioUrl) return Promise.resolve(track.podcastAudioUrl);` ——播客单集自带音频直链，跳过整条网易云查询链。
2. `_scrobble(track, ...)`：播客 track 直接 return，不调网易云听歌打卡。
3. `_playAudioSource()`：howler 的 `format` 嗅探列表加上 `'m4a', 'aac', 'mp4'`（小宇宙音频是 m4a/audio-mp4）。
4. 新增入口方法 `playPodcastEpisode(episode, podcastTitle)`：把单集对象包装成 howler 引擎能用的"track 形状"（id 用 `pod:xxx` 前缀避开网易云数字 id；al.picUrl 用单集封面；dt 用秒级 duration × 1000；扩展字段 `podcastAudioUrl` / `podcastEpisodeId`），设到 `_currentTrack`，然后调 `_replaceCurrentTrackAudio` 走原管道——既不查 id，也不查歌词，也不查 lastfm。`_list` 只放一项，播完即停。

**RSS 字段映射要点**（rssParser.js）
- 频道 id = feedUrl（稳定 + 唯一）
- 单集 id = `${feedUrl}::${guid}`（避免不同节目 guid 撞车）
- 频道封面：优先 `<itunes:image href>`，回退 `<image>` 文本
- 单集封面：优先 `<itunes:image href>`，回退频道封面
- 单集时长：`<itunes:duration>` 支持 `HH:MM:SS` / `MM:SS` / 纯秒数
- pubTime（Date.parse 数值）单独存，方便建 Dexie 索引按时间排序

## 6. 踩坑记录 / 注意事项

- ⚠️ Node 版本：项目要求 14/16，系统装的是 24，必须用 nvm 切到 16 再安装依赖，否则原生模块（如 sass、electron 相关）可能编译失败。
- ⚠️ 开发用 Electron 模式（`yarn electron:serve`）：播客要抓第三方 RSS/音频，浏览器模式有 CORS（跨域）限制，Electron 更自由。
- ⚠️ **单实例锁 + 共用数据目录**：原版 YesPlayMusic 同一时间只允许一个实例，且开发版与正式版默认共用 `%APPDATA%\yesplaymusic`。已通过给 fork 设独立 `app.setName/setPath` 解决（见 §5）。这样既能与正式版对照组同时运行，登录/设置/缓存也互相隔离。
- ⚠️ 端口：本 fork 用 `10755`（网易云API）/`27233`（内部 express）/`20201`（dev server）；正式版用 `10754`/`27232`。改端口时这几处要同步：`services.js`、`background.js`(代理+listen+loadURL)、`.env`。
- ⚠️ **IndexedDB 与 dev server 端口强绑定（重要）**：浏览器/Electron 的 IndexedDB 数据按 origin (`host+port`) 隔离。dev server 默认端口 20201，但若上一次 `yarn electron:serve` 没干净退出导致 20201 被残留进程占住，webpack-dev-server v3 会**悄悄换端口**（20202…），新端口的 IndexedDB 是空的，看起来"数据丢了"——实际上原数据还在 `%APPDATA%\YesPlayMusicPodcast\IndexedDB\http_localhost_20201.indexeddb.leveldb`。
  - **治标**：每次启动 dev 前先 `Get-Process electron, node | Stop-Process -Force` 清残留
  - **治本（已做）**：在 `vue.config.js` 的 `devServer` 加 `host: 'localhost'` 并锁定 `port`，被占就 fail，避免悄悄换端口

### 2026-06-07 第九轮迭代：S1 播放彻底排查 + Tier 1（已完成）

**根因诊断**（找到 3 个独立 bug）

**bug-A 并发竞争（"两条音频"）**
- `_playAudioSource` async 开头 `await fade 180ms`
- 用户在 180ms 内切别集 → 两个 `_playAudioSource` 同时跑 → 两个都 fade 完 → 两个分别 `Howler.unload()` + `new Howl` + `play()` → 两条音频同时响
- **修**：加 `_playSourceToken` 自增令牌，每个调用 capture `myToken`；await 结束后比对当前 `_playSourceToken`，不一致就丢弃旧的调用，旧 howler 不创建/不播放

**bug-B seek 注册到旧 howler（"切回来重播 / 进度失效"）**
- `_replaceCurrentTrackAudio` 调 `_playAudioSource` **没 await**
- `_loadCurrentPodcastEpisode` 拿到 `replaced=true` 时 `this._howler` 还是旧的（fade 中）
- 它在旧 howler 上注册 `once('load', seekTo)` —— 永远不触发新 howler 的 load → seek 不跑 → 用户感觉从头播
- **修**：① `_replaceCurrentTrackAudio` 改 `async` 链 `await _playAudioSource`；② seek 逻辑下沉到 `_playAudioSource` 内部，加 `seekToOnLoad` 参数，避免外部注册时机错；③ once 顺序保证 seek 先于 play

**bug-C 加载条在进度条上方而非同行**
- `.buffering-bar` 用 `top: 0`，但 `.progress-bar` 有 `margin-top: -6px`，错位
- **修**：`top: 50%; transform: translate(X, -50%)`，与 vue-slider 同高度居中

**附加改进**
- `playPodcastEpisode` 入口立即 `setAudioBuffering(true) + setTitle()`：用户点一下立刻看到加载状态 + 新封面，不等异步链
- 之前 progress setter 拖动立即写库已经做了（bug 2 上次修过）

#### B-29 播客库丝滑缩放 + 窗口缩放无极过渡（2026-06-07 用户新增 / B）
**两类缩放并存，互不冲突**

**A 类：Ctrl + 鼠标滚轮 → 用户主动调整封面大小**
- 滚轮向下 → 缩小封面（最小：所有节目一屏显示完）
- 滚轮向上 → 恢复（**不允许放大超过初始尺寸**，明确限制）
- 排列方式不变（按现有顺序），只改单个卡片尺寸
- 用 CSS `--card-size` 变量实现，配合 `grid-template-columns: repeat(auto-fill, minmax(var(--card-size), 1fr))`
- 配合 `transition` 让单卡片尺寸变化时无跳变

**B 类：拖动窗口边缘 → 系统级窗口大小变化**
- 当前体验差：列数从 4→5 时**卡片尺寸突变 + 第 5 列突然出现** = 两个跳变叠加
- **建议算法**（参考成熟方案）：
  - **方案 1**：固定列数（如 5 列），卡片宽度 = `(容器宽 - 间距) / 5`，窗口拖动时卡片**平滑同步缩放**，不增减列数。用户想看更多列就用 Ctrl + 滚轮（A 类）。
  - **方案 2**：列数仍随窗口变化，但卡片用 `view transitions` API 或 FLIP 动画平滑插值（IE 不支持，但 Electron Chromium 支持）
- **建议选方案 1**：行为最可预测，无依赖，最稳。"是否要变列数"完全交给用户主动操作（A 类）
- **副作用**：用户调小窗口时如果卡片小到不可读，列数也不变 → 这时候用 min-width 强制最小尺寸 + 超出滚动

**实施清单**
- 在 `podcastLibrary.vue` 加 `data: { cardSize: 180 }`（默认 180px）
- 监听容器 `@wheel.ctrl.prevent="onZoom"` 调整 cardSize

## B-30 round | S 级回放 bug 修复 + 真实收听时长统计系统（2026-06-07）

### 起因 / 现象
- **S 级 bug**：单集只要被标记为「已听完」（saved progress ≈ duration），用户再点播放，会立刻触发 howler 的 `onend`，跳到下一集，无法重听。
- 没有"真实听过多少秒"的概念。现在的进度只是「最近播到第 N 秒」，无法分辨：用户是从头听到尾、还是只是把进度条拖到末尾。
- 没有累计收听时长（墙钟时间 / 按倍速换算的内容时间），后面"我的最爱播客 = 听得最多的播客"没数据可用。

### 修复 / 新增

1. **S-bug 修：`src/utils/Player.js _loadCurrentPodcastEpisode`**
   - 加载单集进度后，若 `savedPos > 总时长 - 30s`，说明上次已经听完了 → 这次重听强制 `savedPos = 0`，并清掉旧 `episodeProgress` + 重置 listenStats 的 bits（保留累计时长）。
   - 同时 `this._lastListenSec = -1`，避免跨集的"上一秒"污染。

2. **新表 `episodeListenStats`（Dexie v7，`src/utils/db.js`）**
   ```
   episodeListenStats: '&id, completed, updatedAt'
   ```
   行结构：
   - `id`：episode.id
   - `totalSec`：单集总时长（秒）
   - `bits`：`Uint8Array`，按位记录"第 i 秒被听过"（1 bit/秒，30 分钟节目 ≈ 225 字节）
   - `listenedSec`：bits 中被置 1 的总数，即**去重的真实听过秒数**
   - `completed`：`listenedSec >= totalSec - 5`（留 5s 容差）
   - `totalPlayWallSec`：累计墙钟秒数（用户真花的时间）
   - `totalPlayContentSec`：按倍速换算的内容时间秒数（1x → +1，1.5x → +1.5）

3. **`src/utils/podcast/listening.js`（新）**
   - `tickListen(id, sec, totalSec, { recordBit, wallDeltaSec, contentDeltaSec })`：每秒调一次。
   - `resetEpisodeListening(id)`：清 bits/listenedSec/completed，**保留** 两个累计时长。
   - `getListenStats(id) / getAllListenStats()`：UI 层取数据。
   - `listenedPercentStepped(row)`：百分比 floor 到 5 步进；<5% 返 0（不显示标签）。

4. **`Player.js` setInterval 改造（每秒 tick）**
   - `sec = floor(seek)`，`lastSec = this._lastListenSec`
   - `bigJumpTh = max(45, totalSec * 0.3)`，`jump > bigJumpTh` → 视为"用户大幅拖动"
   - 大跳：`recordBit=false, contentDeltaSec=0`（不计入听过秒、不计入倍速内容秒）
     - 但 `wallDeltaSec=1` 仍记墙钟，因为"用户确实在这一秒花了时间"
   - 正常 / +30s / -15s：`recordBit=true, contentDeltaSec=rate`（按倍速累内容秒）

5. **UI 标签：`podcastDetail.vue` / `episodeDetail.vue`**
   - 已听完 → 绿色「已听完」（`#27ae60`）
   - 5%~95% → 黄色「听过 X%」（`#e0a800`）
   - <5% 且没听完 → 不显示（或显示剩余时长，episodeDetail 保留了旧逻辑）

### 设计抉择
- **为什么 bits 而不是区间数组**：节目时长固定，bit 操作 O(1)，去重天然成立，存储 ~225B/集，远小于区间合并方案。
- **为什么 5% 步进**：避免「听过 37%」这种不稳定的小变化让 UI 抖。
- **为什么 totalPlayWallSec 不在 reset 时清掉**：这是"花在这集上的总时间"，重听只会让它累加，永远不该回退。后续"最爱播客"排行用的就是它。
- **为什么不另起一个 DB**：Dexie 同一个 DB 里加表零成本，version bump 即可；分 DB 反而要管两套连接。

### 测试要点
- 已听完的单集再点"立即播放" → 应能从头放，不再立即跳下一集 ✓
- 拖进度条把整集滑到末尾 → 不能解锁「已听完」（因为 bits 是稀疏的）✓
- +30s 连按 → 计入 listenedSec（recordBit=true）✓
- 大拖（一次拖过 30% 时长）→ 不计 listenedSec、不计 content sec ✓
- 倍速 1.5x 听 60s → wallSec +60，contentSec +90 ✓


### B-30 round 续：订阅页累计收听时长 UI

- **`listening.js` 新增 `getPodcastListenSummary(podcastId)`**：用 `db.episodes.where('podcastId').equals(id)` 查到该档所有 episode.id，再 `bulkGet` listenStats 聚合。返回 `{ wallSec, contentSec, finishedCount, totalCount }`。

### 4) ⭐ 下载彻底修复（`electron/podcastDownload.js` 重写为 Node 原生 https）
两个致命 bug 串联，逐个定位：
- **Bug A：`AbortController is not defined`**
  - Electron 13 主进程 Node 是 **v14.16.0**，没有全局 `AbortController`（Node 15+ 才有）。
  - 先改用 axios `CancelToken`，过了这关。
- **Bug B：`socket disconnected before secure TLS connection`（代理环境）**
  - 用户网络是 **Clash Verge TUN 模式**，且 shell 有 `HTTPS_PROXY=http://127.0.0.1:7897` 环境变量。
  - axios 0.26 的 `proxy:false` 只作用于首个请求，**follow-redirects 处理 302 重定向时仍读 `HTTPS_PROXY`**，
    对重定向后的 https 目标做 http-proxy 隧道握手失败。
  - 音频 url 是 `dts-api.xiaoyuzhoufm.com/.../media.xyzcdn.net/....m4a` —— dts-api 302 重定向到 media.xyzcdn.net。
  - 用 `curl --noproxy` 直连和走代理都成功（http 200）→ 证明正确做法是「直连，交给 TUN 网卡层路由」。
  - **终极方案**：弃用 axios，改 **Node 原生 `https`/`http` + 手动跟随重定向**。`https.get` 默认不读代理环境变量，
    天然直连。TUN 模式下流量被网卡层按规则路由（国内 DIRECT、国外代理）；不开代理时直连国内 CDN 也 OK。
    → **开/不开代理都能下**，把代理决策完全交给 TUN。
  - `streamGet(url)` 递归跟随 3xx location（≤6 次），resolve 最终 200 响应流；`req.setTimeout(30s)` 防卡死。
  - 取消：`activeTasks` 存 `{ res, writeStream, filePath, canceled }`，cancel 时 `res.destroy()` + `writeStream.destroy()` + unlink。

### 4) Scrapling 爬虫评估（结论：当前不采用）
- Scrapling 是 **Python** 自适应爬虫框架，三档 Fetcher：`Fetcher`(HTTP+TLS指纹,轻) / `DynamicFetcher`(Playwright Chromium) /
  `StealthyFetcher`(Camoufox Firefox, 绕 Cloudflare Turnstile, 最重)。防反爬很强。
- **不适合当前项目**：
  - 语言不匹配——本项目 Electron/Node，Scrapling 是 Python，集成要额外装 Python + 浏览器内核 + 子进程 IPC，成本高。

- **建议**：首页发现页用主进程 axios（带 UA+Referer）抓 JSON 接口即可。仅当未来遇到 Cloudflare 强防护 / 必须 JS 渲染的源，
  才考虑把 Scrapling 作为**独立 Python 微服务**（本地起一个端口，app 通过 HTTP 调用），与主程序解耦，避免拖累 app 性能。

### 1) ⭐ 订阅自动更新 D-3（我的订阅页"刷新"）

**根因 / 动机**：此前"更新单集"只能靠用户重新粘同一 RSS（B-37 记过）。数据其实早已就绪——每档 podcast 存了 `id=feedUrl`，`upsertEpisodes` 用 `bulkPut`（按 `episode.id=${feedUrl}::${guid}` 自然 upsert），旧集元数据刷新、新集插入，且**进度在 episodeProgress 独立表、不受影响**。所以"刷新"= 对每档重抓 RSS→解析→bulkPut + diff 出新增数。

**关键实现**（`utils/podcast/service.js` 新增 `refreshAllSubscriptions`）：
- 并发限流 `runLimited(items, 5, worker)`：N 个 runner 各自从队列 `shift` 取任务直到取空。**为什么限 5**：RSS 站点(小宇宙 feed.xyzfm.space 等)对突发并发敏感，18 档全并发易被限流/超时；5 路兼顾速度与稳定。

- 单档失败(网络/解析)`catch` 记入 `results[].error`，**不阻塞**其它档。返回 `{ totalNew, results }`。

**新单集角标（持久 + 可清）**：
- `db.js` 新增 `updatePodcast(id, patch) = db.podcasts.update(...)`。**关键点**：`newCount` 不是 Dexie 索引字段，可直接存，**无需升 schema 版本**；且用 `.update` 局部改、不动 `updatedAt`（否则"按订阅时间"排序会被打乱）。
